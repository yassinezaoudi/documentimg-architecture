## Домашнее задание 01: Документирование архитектуры в Structurizr

### Вариант 17: Система управления рестораном (аналог OpenTable)

**Описание системы**  
Необходимо разработать систему для управления рестораном, которая позволяет клиентам просматривать меню, создавать заказы, а персоналу ресторана управлять меню и статусами заказов. Система должна предоставлять API для работы с пользователями, блюдами и заказами.

### 1. Роли пользователей и внешние системы

**Роли пользователей**  
- **Customer (Клиент)** – посетитель ресторана, который регистрируется, просматривает меню, создаёт заказы и отслеживает их статус.  
- **Staff (Сотрудник ресторана)** – администратор или менеджер, который управляет меню (добавление/изменение блюд), просматривает все заказы и изменяет их статус.

**Внешние системы**  
- **Payment Gateway** – внешняя платёжная система (например, Stripe) для обработки оплаты заказов.  
- **Email Service** – внешний сервис для отправки email-уведомлений (например, SendGrid) о статусе заказа.

### 2. Основные задачи пользователей (Use Cases)

**Customer**  
- Зарегистрироваться / найти себя по логину или маске имени/фамилии.  
- Просматривать меню (все блюда или по категориям).  
- Создавать заказ, добавлять в него блюда.  
- Просматривать историю своих заказов.

**Staff**  
- Добавлять новые блюда в меню.  
- Просматривать все заказы.  
- Изменять статус заказа (например, «принят», «готовится», «доставлен», «оплачен»).

### 3. Перечень контейнеров

Для реализации системы выбрана микросервисная архитектура. Каждый контейнер отвечает за ограниченный контекст предметной области.

| Контейнер           | Технологии              | Описание                                                                 |
|---------------------|-------------------------|--------------------------------------------------------------------------|
| Web Application     | React, HTML, CSS        | Одностраничное приложение для клиентов и персонала.                      |
| API Gateway         | Spring Cloud Gateway    | Единая точка входа, маршрутизирует запросы к микросервисам.              |
| User Service        | Spring Boot (REST API)  | Управление пользователями (регистрация, поиск).                          |
| Menu Service        | Spring Boot (REST API)  | Управление меню (блюда, категории).                                      |
| Order Service       | Spring Boot (REST API)  | Управление заказами (создание, добавление блюд, изменение статуса).      |
| User Database       | PostgreSQL              | Хранит данные пользователей.                                             |
| Menu Database       | PostgreSQL              | Хранит данные о блюдах и категориях.                                    |
| Order Database      | PostgreSQL              | Хранит данные о заказах и составе заказов.                              |

### 4. Взаимодействие между контейнерами

- **Пользователи** взаимодействуют с **Web Application** через браузер (HTTPS).  
- **Web Application** отправляет API-запросы в **API Gateway** (HTTPS/REST).  
- **API Gateway** маршрутизирует запросы к соответствующим микросервисам:
  - `/users` → **User Service**
  - `/menu` → **Menu Service**
  - `/orders` → **Order Service**
- Каждый микросервис работает со своей базой данных через **JDBC**.
- Для выполнения бизнес-сценариев **Order Service** может обращаться к **User Service** и **Menu Service** для получения дополнительной информации (проверка существования пользователя, получение цены блюда).
- **Order Service** также взаимодействует с внешними системами: инициирует платежи через **Payment Gateway** и отправляет уведомления через **Email Service**.

### 5. Динамическая диаграмма для сценария «Создание заказа»

Архитектурно значимый сценарий: клиент создаёт новый заказ.  
На диаграмме показана последовательность взаимодействия между контейнерами.

1. **Customer** нажимает кнопку «Оформить заказ» в веб-приложении.  
2. **Web Application** отправляет `POST /orders` с идентификатором пользователя и списком блюд в **API Gateway**.  
3. **API Gateway** перенаправляет запрос в **Order Service**.  
4. **Order Service** запрашивает данные пользователя у **User Service** (`GET /users/{userId}`).  
5. **User Service** выполняет запрос к **User Database** и возвращает информацию.  
6. **Order Service** для каждого блюда запрашивает его цену у **Menu Service** (`GET /dishes?ids=...`).  
7. **Menu Service** получает данные из **Menu Database** и возвращает их.  
8. **Order Service** создаёт запись заказа в **Order Database** (таблицы `orders` и `order_items`).  
9. **Order Service** инициирует платёж через внешний **Payment Gateway**.  
10. После успешного создания платежа **Order Service** отправляет email-уведомление через **Email Service**.  
11. **Order Service** возвращает подтверждение (созданный заказ) в **API Gateway**.  
12. **API Gateway** передаёт ответ в **Web Application**.  
13. **Web Application** отображает клиенту подтверждение заказа.

Все межсервисные вызовы выполняются по протоколу **HTTPS/REST**, обращения к базам данных – через **JDBC**.

Модель архитектуры, включая диаграммы System Context, Container и Dynamic, описана на языке Structurizr DSL в файле [`workspace.dsl`](workspace.dsl).
